Tree Shaking Contract Specs

How we taught the Rust compiler to tell us which types a smart contract actually exposes — by hiding breadcrumbs in the WASM data section and letting dead-code elimination do the rest.

Hacking Rust to Build Contracts01
The artifact

A contract is code plus a description of itself

A Soroban contract compiles to a single .wasm. Tucked inside a custom section, contractspecv0, is the contract's interface spec (SEP-48): every function signature, user type, error, and event — serialized as XDR (Stellar's binary encoding).

Tooling lives on that spec: the CLI, client bindings in other languages, explorers, wallets. It's how the outside world knows how to call your contract.

The artifact02
The problem

Every type you define ends up in the spec —
even the ones you never expose

  • Define it, ship it. Each #[contracttype], #[contracterror], #[contractevent] emits a spec entry — whether or not it appears at the contract's boundary.
  • Internal types leak. Storage structs, helper enums, intermediate shapes — all described to the world.
  • Imports drag in everything. contractimport! carries an interface's whole type set — with no reachability there's no way to keep only what you actually call.
  • Bigger = worse. On-chain bytes cost rent; a bloated spec is dead weight in every download, every explorer, every binding.

The spec should be the contract's public surface — not a memory dump of every type the author happened to declare.

The problem03
Spec shaking v1 — the workaround we lived with

Guess from pub and a manual export flag

the heuristic
  • pub types → exported
  • non-pub types → hidden
  • all events, all functions → exported
  • contractimport! (wasm) → export nothing
  • override manually: export = true|false
why it hurt
  • Visibility ≠ interface. A pub helper is still in your API surface.
  • All-or-nothing imports. Re-exposing an imported type meant re-exposing the whole interface — so imports exposed nothing.
  • Manual bookkeeping. export flags drift out of sync with real usage.

We wanted the compiler to answer one question automatically: is this type actually reachable from the contract's boundary?

Spec shaking v104
The goal

Tree shaking — but for on-chain type specs

JavaScript bundlers "tree shake": start from what's reachable, drop the rest. We want the same for the contract spec.

Keep a type's spec entry if — and only if — it's reachable from a public contract function, a published event, or a thrown error. Strip everything else. Automatically. No flags.

The goal05
Why this is hard #1

Proc-macros are blind to the whole program

A derive macro sees exactly one item at a time. When #[contracttype] expands over struct Foo, it has no idea whether Foo is used at a boundary somewhere else in the crate.

Macros run in isolation, in unspecified order, cache-independently. There's no shared state, no second pass, no "ask me again once you've seen everything."

So the macro cannot decide "should this be in the spec?" at expansion time.

lib.rs
#[contracttype]       // expands here…
pub struct Foo { pub x: u32 }

// …but "is Foo used?" is decided
// 300 lines away — in another file,
// another macro invocation the
// compiler runs independently.
#[contractimpl]
impl Contract {
    pub fn go(e: Env, f: Foo) { … }
}
Why it's hard · macros06
Why this is hard #2

Type identity can't ride on the name

  • Names collide. Two modules define Status. A local Balance and an imported Balance are different specs with the same label.
  • Names are lossy. The spec entry records structure, doc, fields — the name alone can't tell one shape from another.
  • We need a stable fingerprint that survives compilation and means "this exact spec entry," independent of what it's called.

identity(entry) = SHA-256( entry.to_xdr() )▊

Hash the serialized spec entry, not its name. Same bytes on both sides — inside the WASM and in the post-build tool.

Why it's hard · identity07
The insight

Something already computes whole-program reachability

The part of the pipeline that runs after the macros and already computes reachability for free is the compiler + linker's dead-code elimination.

If unreachable code gets stripped anyway… then attach a breadcrumb to each type, and let DCE tell us which ones survive.

We can't ask the macro. And the spec is an opaque contractspecv0 blob the linker never reads — but DCE does strip unused code and the data it references. So we emit code whose survival is a proxy for use, and read the answer out of the finished binary.

The marker09
The hack · anchoring the breadcrumb

A function whose survival is the signal

generated by #[contracttype]
impl SpecShakingMarker for Foo {
  #[inline(always)]
  fn spec_shaking_marker() {
    #[cfg(target_family = "wasm")] {
      static MARKER: [u8; 14] =
        *b"SpEcV1\x93\xd5\x05\x04\x36\x68\x40\xd2";
      let _ = unsafe {
        read_volatile(MARKER.as_ptr()) // anchor
      };
    }
  }
}
  • Trait with a no-op default. Primitives & SDK types do nothing; contract types get this impl.
  • read_volatile forbids the optimizer from folding the static away — if the function runs.
  • #[cfg(wasm)] — native/test builds pay nothing; markers exist only in the shipped artifact.
  • One byte is enough. Volatile-touching the symbol anchors all 14; reading more just grows code.

Boundaries11
The hack · nesting & recursion

Markers that reach through types

  • Fields chain. A struct's marker fn calls its field types' marker fns — one boundary hit marks the whole tree.
  • Inline shapes call directly. Option<T>, Result<T,E>, tuples — their size includes the inner type, so a direct call is safe.
  • Indirection uses keep_reachable. Vec<T>, Map<K,V>, &T are pointer-sized regardless of T → a recursive type could loop forever. So we take the fn pointer and volatile-read it, never calling it.
spec_shaking.rs
// direct — inner is inline
impl<T: SSM> SSM for Option<T> {
  fn marker() { T::marker(); }
}

// indirect — break the cycle
impl<T: SSM> SSM for Vec<T> {
  fn marker() {
    keep_reachable(T::marker);
  }              // ↑ fn ptr, not a call
}

Post-build filter14
Opt-in signal

How does the tool know to shake?

A WASM built the old way has no markers. Shaking it blindly would strip every type. So v2 writes a version note in contractmetav0:

rssdk_spec_shaking = "2"

Absent or "1" → the tool leaves the spec untouched. "2" → markers are present, safe to shake.

  • Forward-safe. Old tools ignore an unknown meta key; old WASMs are never mis-shaken by new tools.
  • Version-negotiated. The meta value is the protocol between SDK and CLI — bump it, evolve the scheme.
  • Self-declaring. Anyone inspecting the WASM can see exactly how its spec was produced.

The payoff

Specs that describe the interface, nothing more

only used

Types, errors & events survive only when reachable from a boundary — decided by the compiler, not by hand.

self-contained

Imported types used at the boundary are now included & kept; unused imports vanish. Import safely at last.

zero flags

No export bookkeeping. Visibility no longer leaks into your public API.

Same source, same macros — the difference is entirely in what the linker keeps alive and what the CLI reads back out.

The payoff16
Shipping it · the hard part

The clever part was easy. Rolling it out was not.

v2 makes the SDK export everything, trusting the CLI to strip the rest. Build with plain cargo build and no CLI runs… so your spec gets bigger, not smaller.

  • The optimization is split across two tools: the SDK plants markers, the CLI harvests them.
  • Ship the SDK first and half the pipeline is missing — a strict regression for anyone not on the new CLI.
  • A compile-time feature can't detect what build tool is driving it… unless we make it.
Rollout · chicken & egg17
Shipping it · the guard

Make the build fail without the right tool

soroban-sdk/build.rs
// feature on + wasm target…
if feature_enabled && target_is_wasm {
  if env::var(
    "…SUPPORTS_SPEC_SHAKING_V2"
  ).is_err() {
    eprintln!("error: requires \
       stellar-cli v25.2.0+");
    process::exit(1);
  }
}
stellar-cli sets it
cmd.env("…SUPPORTS_SPEC_SHAKING_V2", "1");
  • The CLI is the handshake. Only stellar contract build sets the env var → the SDK trusts the harvest step will run.
  • Fail loud, fail early. A wrong build errors at compile time with a fix, instead of silently shipping a bloated spec.
  • Native builds untouched. The guard only fires for wasm; your unit tests never see it.

The long tail

Where "reachable" got subtle

#1852

Panic-only errors

An error thrown via panic_with_error! never appears in a Result return — so it needs its own boundary marker, or it'd be shaken out.

#1835

Recursive types

A type reachable through Vec<Self> would recurse forever on a direct call → the keep_reachable fn-pointer trick.

#1876

Deprecating export

Under v2, export is a no-op — worse, it can drop a marker for a reachable type. Now it warns and will be removed.

Each is a small hole in the "reachability = used" model. The pattern held; the edges just needed their own roots.

What's next

From markers-per-type to roots + a graph

Marker-per-type still leans on hashing to reconnect types. The next iteration (PR #1861) makes identity exact:

  • Functions are roots — read straight from the public spec.
  • Markers only for hidden roots — events & thrown errors, which don't show up in a function signature.
  • A removable sidecar graph (SpGrV1) records exact spec-entry references. Traverse it; keep reachable types.
  • Strip the graph from the final WASM — the scaffolding never ships.

Takeaways

A trick you can steal

  • Let the compiler and linker answer what macros can't. When you need whole-program reachability but only have per-item metaprogramming, encode a proxy the compiler will keep-or-kill, then read it back from the artifact.
  • Identify by content hash, not by name. Names are ambiguous across modules and imports; a hash of the serialized thing is stable on both sides of the pipeline.
  • A side channel in the binary is a protocol. Data-section breadcrumbs + a version note in meta = a contract between compiler and tooling.
  • Shipping is the feature. Feature flags, build guards, and a staged protocol rollout mattered as much as the hack itself.
